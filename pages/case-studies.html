Example 2 · Databricks startup overhead

A $50K opportunity hidden in billing data

Stable Databricks jobs were masking a costly startup-time anomaly.

  • Some clusters were burning ~20% more DBUs per job than peers with the same setup.
  • Billing patterns showed spend starting several minutes before any real work began.
  • Engineering traced the waste to thousands of tiny files scanned at cluster startup.
  • Compacting files cut startup from 8 min to 2 min and reduced Databricks cost ~18%.

Context: daily jobs supported property valuation and lease forecasting while the bill kept climbing despite steady usage.

Example 3 · Data residency inefficiency

Hidden local-disk storage driving avoidable spend

CloudDIET exposed storage waste that native Azure tools were missing.

  • Storage cost kept rising even while data volume and compute usage stayed steady.
  • Spend was shifting from ADLS meters to VM-attached local disks on Databricks clusters.
  • Temporary parquet files and cached tables were written to local paths and left behind.
  • Redirecting outputs to ADLS and cleaning up managed disks removed ~$45K/year of avoidable spend.

Root cause & fix

Root cause: ETL notebooks were writing temp parquet files to local /dbfs paths instead of ADLS.

Fix: re-point outputs to ADLS, enable cache cleanup, and remove unused managed disks.

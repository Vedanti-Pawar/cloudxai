Service-specific engineering intelligence

Azure services don't optimize the same way

CloudDIET profiles billing, configuration, and usage so it can apply the right optimization logic to each service family.

Azure Cosmos DB

  • Capacity choices: Provisioned, Autoscale, serverless, burst, and vCore models.
  • API & data-model differences: SQL, Mongo, Cassandra, Graph, PostgreSQL, and Table.
  • Topology levers: replicas, partitions, consistency, and indexing.

Azure App Services

  • Plan tiers & eligibility: Consumption, Shared, Basic, Standard, Premium, and Isolated.
  • Runtime shape: Windows vs Linux, Functions or Web Apps, integrations and registries.
  • Scaling behavior: autoscale rules, fixed scale, allocated plan vs actual use.

Event Hubs / Messaging

  • SKU fit: Basic, Standard, Premium, and Dedicated capacity models.
  • Throughput design: partitions, publisher-consumer activity, and performance trade-offs.
  • Elasticity: scale-out and scale-back patterns across messaging workloads.

Azure Databricks

  • Compute efficiency: idle clusters, zombie compute, and dev-test DBUs running 24×7.
  • Storage behavior: local-disk data residency, governance gaps, and ADLS alignment.
  • Commercial signals: Savings Plan or RI fit, workload forecasting, and commit sizing.
CloudDIET uses deep, service-aware logic to find savings generic reporting tools miss.
Representative measures

What CloudDIET evaluates, by service family

Cosmos DB

  • Write-replica regions used only for reads, or not at all
  • Instances with no data, or data but no reads/writes
  • Hot partitions, caching fit, indexing-consistency tuning

App Services

  • Consumption vs allocated spend by plan type
  • Misconfigured scale-out and fixed-scale plans without autoscale
  • Unused integration accounts and inactive registries

Messaging

  • Overprovisioned hubs/buses with no publisher or consumer activity
  • Partition sizing and throughput optimization
  • Standard vs Premium vs Dedicated SKU fit

Databricks

  • Idle or underutilized clusters and dev-test DBUs running continuously
  • Ungoverned storage and local-disk data residency
  • Commit-smarter guidance for DBU pre-purchase
Customer case study

Databricks startup overhead waste

  • Billing pattern: some clusters burned ~20% more DBUs per job before real work began.
  • Root cause: upstream pipelines created too many small files, increasing startup scan time.
  • Fix: merge files into larger chunks to cut wasted startup overhead.
  • Result: startup fell from 8 min to 2 min — ~18% lower Databricks cost, ~$50K saved annually.
